<a id="webapi-order-backup-get"></a>


| | Web API / 主接口(Rest API) / 交易 / 订单 / 从备份获取订单 | |
| --- | --- | --- |






从备份获取订单


该请求可以从服务器上的特定备份接收一个或多个订单的信息。


请求格式


```
GET /api/order/backup/get?backup=date&login=login&ticket=ticket&from=beginning&to=end&server=identifier

POST /api/order/backup/get?backup=date&login=login&ticket=ticket&from=beginning&to=end&server=identifier
```


响应格式


> { "retcode" : "code description", "answer" : [ description of orders ] }


示例


```
//--- 发送到服务器的请求

GET /api/order/backup/get?backup=1574122620&login=104366

//--- 服务器响应

{

 "retcode" : "0 Done",

 "answer" : [

 {

 "Order" : "12832917",

 "ExternalID" : "",

 "Login" : "104366",

 ...

 },

 {

 "Order" : "12832918",

 "ExternalID" : "",

 "Login" : "104366",

 ...

 },

...

 ]

}
```


请求参数


- backup — 备份副本日期。要获取可用备份的列表，请使用/api/order/backup/list请求。

- server — 备份服务器标识符，从这里请求订单。可选参数。如果没有指定，则将从列表中的第一个备份服务器请求数据。

- login — 应从备份数据库检索其订单的用户的登录名。如果请求中没有指定订单单号，则需要这个参数。

- ticket — 您希望从备份数据库获取的订单单号。

- from — 请求订单的时段的开始。该日期从1970.01.01开始，以秒数指定。这个参数不可以与'ticket'参数一起使用。

- from — 请求订单的时段的结束。该日期从1970.01.01开始，以秒数指定。这个参数不可以与'ticket'参数一起使用。


响应参数


- retcode — 如果成功，该命令返回响应代码 0。否则，将返回错误代码。

- answer — JSON格式的订单参数。传递订单参数的完整描述提供在“数据结构”部分。

| | Web API / 主接口(Rest API) / 交易 / 订单 / 获取备份列表 | |
| --- | --- | --- |






获取备份列表


该请求可以接收指定时间范围内订单备份创建日期。


请求格式


```
GET /api/order/backup/list?from=beginning&to=end&server=identifier

POST /api/order/backup/list?from=beginning&to=end&server=identifier
```


响应格式


> { "retcode" : "code description", "answer" : [ date list ] }


示例


```
//--- 发送到服务器的请求

GET /api/order/backup/list?from=1546357749&to=1574351349

//--- 服务器响应

{

 "retcode" : "0 Done",

 "answer" : [ 1574122620,1574209020,1574250008 ]

}
```


请求参数


- from — 希望接收备份列表的时段的开始。该日期以自 1970 年 1 月 1 日以来经过的秒数指定。

- to — 希望接收备份列表的时段的结束。该日期以自 1970 年 1 月 1 日以来经过的秒数指定。

- server — 备份服务器标识符，从这里请求备份。可选参数。如果没有指定，则将从列表中的第一个备份服务器请求副本。


响应参数


- retcode — 如果成功，该命令返回响应代码 0。否则，将返回错误代码。

- answer — 在服务器上有备份的日期列表。该日期以自 1970 年 1 月 1 日以来经过的秒数指定。


注意


接收的日期可被用于[/api/order/backup/get](webapi_order_backup_get.htm)，以便从特定备份请求订单数据。

<a id="webapi-order-data-structure"></a>


| | Web API / 主接口(Rest API) / 交易 / 订单 / 数据结构 | |
| --- | --- | --- |






数据结构


订单相关信息以 JSON 格式作为响应命令commands [ORDER_GET](webapi_order_get.htm)，[ORDER_GET_PAGE](webapi_order_get_page.htm)， [HISTORY_GET](webapi_history_get.htm)和[HISTORY_GET_PAGE](webapi_history_get_page.htm)的附加主体传递。订单相关信息包含以下参数：


| 选项字段 | 类型 | 描述 |
| --- | --- | --- |
| Order | 整数 | 订单单号。 |
| ExternalID | 字符串 | 外部交易系统中的订单 ID。 |
| Login | 整数 | 订单所属客户的登录。 |
| Dealer | 整数 | 已处理订单的交易员的登录。 |
| Symbol | 字符串 | 订单交易品种。 |
| Digits | 整数 | 订单价格中的小数位数。 |
| DigitsCurrency | 整数 | 已下单的客户入金货币的小数位数。 |
| ContractSize | Float | 下单所针对的交易品种的合约大小。 |
| State | 整数 | 订单的当前状态。传递为EnOrderState枚举值。 |
| Reason | 整数 | 执行交易的原因。传递为EnOrderReason枚举值。 |
| TimeSetup | 整数 | 从 1970 年 1 月 1 日开始经过的秒计下单的时间。 |
| TimeExpiration | 整数 | 从 1970 年 1 月 1 日开始经过的秒计订单到期的时间。 |
| TimeDone | 整数 | 从 1970 年 1 月 1 日开始经过的秒计订单执行的时间。 |
| Type | 整数 | 订单类型。传递为EnOrderType枚举值。 |
| TypeFill | 整数 | 订单成交类型。传递为EnOrderFilling枚举值。 |
| TypeTime | 整数 | 订单到期类型。传递为EnOrderExpiration枚举值。 |
| PriceOrder | Float | 订单价格。 |
| PriceTrigger | Float | 触发止损限价订单时提交限价订单所依的价格。 |
| PriceCurrent | Float | 已下单所针对的交易品种的现行价格。 |
| PriceSL | Float | 订单的止损水平。 |
| PriceTP | Float | 订单的止盈价位。 |
| VolumeInitial | 整数 | 初始订单交易量。一单位相当于 1/10000 手，例如，10500表示1.05手。 |
| VolumeCurrent | 整数 | 订单的当前未成交量。一单位相当于 1/10000 手，例如，10500表示1.05手。 |
| ExpertID | 整数 | 已下单的策略交易系统的 ID。 |
| PositionID | 整数 | 订单中设置的持仓 ID（单号）。 |
| PositionByID | 整数 | 订单中设置的对应持仓 ID（单号）。针对 Close By 操作（OP_CLOSE_BY）设置的属性。PositionID 中设置依据对应持仓平仓的持仓的单号。 |
| Comment | 字符串 | 对订单注释。 |
| ActivationMode | 整数 | 订单激活类型。传递为EnOrderActivation枚举值。 |
| ActivationTime | 整数 | 从 1970 年 1 月 1 日开始经过的秒计订单激活的时间。 |
| ActivationPrice | Float | 已激活订单所依的价格。 |
| ActivationFlags | 整数 | 订单激活标识。传递为EnTradeActivationFlags枚举值（适当标识值的和）。 |
| TimeSetupMsc | 整数 | 毫秒计下单时间。 |
| TimeDoneMsc | 整数 | 毫秒计订单执行时间。 |
| RateMargin | Float | 计算订单的预付款要求时使用的、交易品种预付款货币到用户入金货币的转换比率。 |
| ModificationFlags | 整数 | 订单修改标识。传递为 EnTradeModifyFlags枚举值（适当标识值的和）。 |

删除未结订单


该请求可以根据单号删除一个或多个未结订单。


请求格式


```
GET /api/order/delete?ticket=tickets

POST /api/order/delete?ticket=tickets

POST /api/order/delete

[

 1012,

 4034

]
```


响应格式


> { "retcode" : "code description" }


示例


```
//--- 发送到服务器的请求

GET /api/order/delete?ticket=73339,73340

//--- 服务器响应

{ "retcode" : "0 Done" }
```


请求参数


- ticket — 要删除的订单的单号。可用逗号隔开指定多个单号。单号能够以POST请求体指定为数组。


响应参数


- retcode — 如果成功，该命令返回响应代码 0。否则，将返回错误代码。


注意


- 只能在连接到订单已被创建所在的相同交易服务器时删除订单。如果没有找到指定单号的订单，则将返回响应代码13。

- 若要使用该函数，通过Web API应用程序使用的经理账户需要拥有以下权限：RIGHT_TRADE_DELETE和RIGHT_TRADES_MANAGER。

获取多个未结订单

请求格式

```
 96639,

 13549

 ],

}
```


响应格式


> { "retcode" : "code description", "answer" : [ description of orders ] }


示例


```
//--- 发送到服务器的请求

GET /api/order/get_batch?ticket=12832917,12832918

//--- 服务器响应

{

 "retcode" : "0 Done",

 "answer" : [

 {

 "Order" : "12832917",

 "ExternalID" : "",

 "Login" : "1020",

 ...

 },

 {

 "Order" : "12832918",

 "ExternalID" : "",

 "Login" : "1020",

 ...

 },

...

 ]

}
```


请求参数


- login — 登录名列表，即您希望接收的订单。用逗号分隔的列表。登录名能够以POST请求体指定为数组。

- retcode — 如果成功，该命令返回响应代码 0。否则，将返回错误代码。

<a id="webapi-order-reopen"></a>


| | Web API / 主接口(Rest API) / 交易 / 订单 / 重开订单 | |
| --- | --- | --- |






重开订单


该请求可以重开账户历史中的挂单。


请求格式


> GET /api/order/reopen?ticket=ticket POST /api/order/reopen?ticket=ticket


响应格式


> { "retcode" : "code description", "answer" : { description } }


示例


```
//--- 发送到服务器的请求

GET /api/order/reopen?ticket=12832917

//--- 服务器响应

{

 "retcode" : "0 Done",

 "answer" : {

 "Order" : "12832917",

 "ExternalID" : "",

 "Login" : "104366",

 ...

 }

}
```


请求参数


- ticket — 要重开的订单的单号。


响应参数


- retcode — 如果成功，该命令返回响应代码 0。否则，将返回错误代码。

- answer — JSON格式的已恢复订单参数。订单参数的完整描述提供在数据结构部分。


注意


只可[重新打开Limit、Stop和Stop Limit订单](imtorder_enum.htm#enordertype)。您要重新打开的订单必须存在于客户历史中。


该请求在账户历史中找出订单并将更新其[参数](webapi_order_data_structure.htm)时将其移到未结订单。


- 将订单状态更改为已下单(State)

- 删除执行日期(TimeDone，TimeDoneMsc)

- 删除触发价格(PriceTrigger)

- 将当前交易量(VolumeCurrent)重置为初始交易量(VolumeInitial)

- PositionID归零

- 重置激活标志：ActivationTime、ActivationPrice和ActivationMode


在重开之前触发订单之前，您应该适当地更正客户交易持仓和账户的状态：


- 删除按照订单打开的交易。如果执行了与订单相关的多个交易，请删除这些全部交易。

- 更正客户的持仓(/api/position/fix)。

- 更正客户的结余(/api/user/check_balance)。


在上述操作之后，客户的状态将与订单历史相符。

| PriceOpen | Float | 持仓的加权平均开仓价。根据以下公式计算：(price of deal 1 * volume of deal 1 1 + ... + price of deal N * volume of deal N) / (volume of deal 1 + ... + volume of deal N)。 |
| PriceCurrent | Float | 已开仓的交易持仓所针对的交易品种的现行价格。 |
| PriceSL | Float | 交易持仓的止损水平。 |
| PriceTP | Float | 交易持仓的止盈水平。 |
| Volume | 整数 | 交易持仓的量。一单位相当于 1/10000 手，例如，10500表示1.05手。 |
| Profit | Float | 交易持仓的当前盈利/亏损。 |
| Storage | Float | 入金货币中持仓的库存费大小。 |
| RateProfit | Float | 持仓利润货币到客户组入金货币的汇率。 |
| RateMargin | Float | 持仓预付款货币到客户的入金货币的汇率。 |
| ExpertID | 整数 | 已开仓的策略交易系统的 ID。 |
| ExpertPositionID | 整数 | 持仓 ID。 |
| Comment | 字符串 | 对持仓的注释。 |
| Dealer | 整数 | 已处理开仓订单的交易员的登录。 |
| ActivationMode | 整数 | 持仓激活类型。以EnActivation枚举值传递。 |
| ActivationTime | 整数 | 从 1970 年 1 月 1 日开始经过的秒计持仓激活的时间。 |
| ActivationPrice | Float | 持仓激活价格。 |
| ActivationFlags | 整数 | 持仓激活标识。传递为EnTradeActivationFlags枚举值（适当标识值的和）。 |
| ModificationFlags | 整数 | 持仓修改标识。传递为EnTradeModifyFlags枚举值（适当标识值的和）。完全平仓或部分平仓的交易承得其修改标识。平仓之后，数据库中不再有单独的持仓相关条目。为了防止丢失修改相关信息，标识会被复制到平仓交易中。同时，会为交易设置其他 EnTradeModifyFlags::MODIFY_FLAGS_POSITION 修改标识，意味着标识承自持仓。 |
| Reason | 整数 | 开仓的原因。传递为EnPositionReason枚举值。 |

- offset — 持仓索引，您需要获得的持仓从其开始。编号从 0 开始。

- total — 应该获得的持仓的数目。可以在一个命令中请求的持仓的最大数目是100。


响应参数


- retcode — 如果成功，命令返回响应代码 0。否则，其将返回错误代码。

- answer — JSON格式的持仓数组。传递持仓参数的完整描述提供在“数据结构”部分。


注意


- 该方法允许轻松安排结果持仓的分页输出。首先，您应该使用/api/position/get_total方法获得客户的持仓的总数。定义应在一页上显示的持仓数（由 TOTAL 参数设置）之后，您可以轻松查找每页的 OFFSET 参数。


- 该请求只适用于客户账户的未结持仓。持仓历史记录根据交易历史记录在客户端形成。无法直接获得。

设置持仓数量


该请求可以获得客户的持仓的总数。


请求格式


> GET /api/position/get_total?login=login POST /api/position/get_total?login=login


响应格式


> { "retcode" : "code description", "answer" : { "Total" : "number" } }


示例


```
//--- 发送到服务器的请求

GET /api/position/get_total?login=1020

//--- 服务器响应

{

 "retcode" : "0 Done",

 "answer" : { "Total" : "10" }

}
```


请求参数


- login — 客户的登录名。


响应参数


- retcode — 如果成功，命令返回响应代码 0。否则，其将返回错误代码。

- total — 客户的当前持仓数。


注意


该请求只适用于客户账户的未结持仓。持仓历史记录根据交易历史记录在客户端形成。无法直接获得。

| | Web API / 主接口(Rest API) / 交易 / 持仓 / 持仓更新 | |
| --- | --- | --- |






持仓更新


该请求可以更改服务器上的持仓。


请求格式


> POST /api/position/update { JSON格式的要更改的持仓描述 }


响应格式


> { "retcode" : "0 Done", "answer" : { description } }


示例


```
//--- 发送到服务器的请求

POST /api/position/update

{

 "Position" : "618",

 "ExternalID" : "",

 "Login" : "764636",

 ...

}



//--- 服务器响应

{

 "retcode" : "0 Done",

 "answer" : {

 "Position" : "618",

 "ExternalID" : "",

 "Login" : "764636",

 ...

 }

}
```


请求参数


该请求没有参数。要更改的持仓的描述以JSON格式作为附加主体传递。持仓参数的完整描述提供在[“数据结构”](webapi_position_data_structure.htm)部分。


> 要更改的记录根据单号来指定。 如果没有指定参数值，则不会更改相应的参数。


响应参数


- retcode — 如果成功，该命令返回响应代码 0。否则，将返回错误代码。

- answer — JSON格式的已更新持仓参数。持仓参数的完整描述提供在数据结构部分。


注意


只能在连接到持仓已被创建所在的相同交易服务器时更新持仓。如果没有找到指定单号的持仓，则将返回响应代码[13](retcodes_common.htm)。

持仓


Web API提供了以下请求用于接收客户持仓的相关信息：


| 请求 | 描述 |
| --- | --- |
| /api/position/get | 通过交易品种名称获得客户的持仓。 |
| /api/position/get_total | 获得客户的开仓总数。 |
| /api/position/get_page | 通过客户的登录获得持仓。 |
| /api/position/get_batch | 根据登录名、单号或组列表获取多个持仓的信息。 |
| /api/position/update | 更改服务器上的持仓。 |
| /api/position/delete | 根据单号删除一个或多个持仓。 |
| /api/position/backup/list | 获取指定时间范围内持仓备份创建日期。 |
| /api/position/backup/get | 获取指定时间范围内持仓备份创建日期。 |
| /api/position/backup/restore | 从备份恢复持仓。 |
| /api/position/check | 根据交易历史检查账户持仓的正确性。 |
| /api/position/fix | 根据交易历史更正账户持仓。 |


传递持仓相关数据的格式，在[“数据结构”](webapi_position_data_structure.htm)中描述。
